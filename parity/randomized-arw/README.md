# Seeded ARW differential corpora

This directory contains the Fortran drivers for deterministic randomized
coverage of every currently translated ARW kernel. The inputs are generated by
the safe Rust tool in `tools/arw-corpus-generator` and committed under
`crates/wrf-dynamics/test-data/randomized-arw`.

`scripts/randomized-arw/run-oracles.sh` performs the complete provenance chain:

1. regenerate all inputs and byte-compare them with the committed files;
2. compile the pinned WRF v4.7.1 routines without rewriting their arithmetic;
3. run the Fortran drivers on those regenerated inputs;
4. compare the complete outputs with Fortran-derived goldens; and
5. let the Rust unit tests consume the same inputs and goldens.

Pass `--write-goldens` only after an intentional generator or upstream-version
change. The resulting diff must be reviewed; Rust never writes expected output.

## Corpus version and population

The generator uses an explicitly implemented SplitMix64 sequence. A unit test
pins its first outputs, and the oracle's byte comparison detects any later
generator drift. Each family has its own seed range:

| Routine | Cases | Seed range | Complete output values |
|---|---:|---:|---:|
| `positive_definite_sheet` | 24 | 1,591,201–3,999,968 | 1,654 |
| `positive_definite_slab` | 16 | 2,771,003–4,726,448 | 12,086 |
| `held_suarez_damp` | 12 | 3,901,111–5,604,549 | 19,698 |
| `calc_mu_staggered` | 16 | 5,100,007–7,791,442 | 6,150 |

The 68 cases cover varying small field shapes, non-one and negative memory
origins, domain/tile clipping, all 16 cross-axis column-mass boundary
combinations, signed zero, finite magnitude extremes, active NaN/infinity
inputs, and untouched storage. Small cases keep a first-divergence report
actionable while the aggregate 39,588 outputs exercise many independent lines.

## Input representation

Files are whitespace-delimited integers so both languages consume one exact
representation. The first integer is the case count. Every case then stores its
seed and dimensions/bounds, followed by field values as signed decimal `int32`
representations of raw IEEE-754 binary32 bits. Fortran uses `TRANSFER`; Rust uses
`f32::from_bits`.

Sheet metadata is:

```text
seed west_east_points south_north_points
```

followed by one total per line and then the field in `i, j` storage order. The
three-dimensional drivers use WRF's ordinary 18 inclusive bounds after the
seed:

```text
ids ide jds jde kds kde ims ime jms jme kms kme its ite jts jte kts kte
```

Fields follow in WRF-contiguous `i, k, j` order. Column mass uses the same
metadata and `i, j` ordering.

## Output and exceptional-value policy

Each output line contains:

```text
seed field_name linear_index expected_value
```

Finite values, signed zero, and infinities compare by all 32 bits. `NAN` means
the result must be a NaN, but its payload and sign are not a compatibility
contract because compilers and future devices may propagate them differently.

The production kernels intentionally do not add a full-field finiteness scan
that WRF lacks. Non-finite atmospheric state is invalid diagnostic input, but
the corpus makes current propagation visible: positive-definite lines containing
only NaN, positive infinity, and nonnegative finite values are unchanged;
Held-Suarez and column-mass active non-finite inputs are checked by output class.
This policy avoids both accidental behavior and a hidden hot-path validation
cost.
